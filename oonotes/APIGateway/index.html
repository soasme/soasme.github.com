D. 结论：如何写好网关？
1. 直接使用业界成熟的实现，例如
a) Spring Cloud
b) AWS Lambda, API Gateway, EC2, etc..
2. 如果要自己设计，需要做好解决以下问题：
a) 路由与代理
(1) 路由 API 端点
(2) 能根据 参数，header 等重定向
(3) 请求逻辑执行前后能套上装饰器
b) 业务开发难度低
c) 社区能提供丰富的中间件
d) 有业务接口协议的方案
(1) schema, swagger
(2) documents 自动生成
e) 接口实现能支持平滑迁移
f) 能缓存特定接口的数据
g) 支持大多数常见内部服务的协议
h) 没有异步回调地狱
i) 有熔断的方案防止雪崩
j) 支持限流
k) 支持限速，根据
(1) 用户
(2) 应用
(3) IP
l) ACL
m) API 统计
(1) 调用量统计
(2) 集成监控应用
n) API 分析
o) 会话
(1) OAuth
(2) Session
(3) JWT
(4) 自定义
p) 日志
3. 一次假想的流程
a) 来自终端的请求(设备，浏览器，命令行，监控设施）
b) 数据通过 Internet 到达用户的负载均衡层
c) 请求到达 API Gateway
d) 根据发布环境加载配置，使网关拥有对特定环境的访问权限
e) 根据路由层的设定，寻找请求对应的资源
f) 检测访问速率，如果超速则驳回请求(429)
g) 身份验证与 ACL 鉴权
h) 检测缓存，如果需要的话
i) Proxy || Function
j) Request Reshaping
k) 将 API 调用次数，延迟，错误率等指标发送到统计服务
4. 系统组件
a) 业务函数，日常开发的主体部分，对 Backend API 进行胶合 + 修饰 + 转发；在业务函数中有 async yield 语法糖，能用同步的写法写异步调用；在业务函数中能 Memoize Context。
b) 路由模块，可以派发用户请求至业务函数
c) 错误处理，可以处理错误，记录日志。
d) 文档与BackendAPIClient自动生成，OAI(Swagger) 是个不错的选择
e) 缓存模块，可以使用 CDN 做资源的加速访问
f) 统计模块，可以收集 API 的相关指标
g) 反爬虫模块，可以离线实时分析访问日志判定爬虫并反馈
h) 限速器，可以对 IP/USER 等做限流/封禁
i) Fault Tolerant Component，参考 Hystrix 的设计
(1) Circuit Beaker，可以在后端服务错误率超出阀值时驳回 Backend API 的访问，避免更多报错积压；一段时间后进入 Half-Open 让行少量流量；最后自动恢复。
(2) Bulkhead Isolation，对后端调用进行线程资源隔离，防止一个应用拖垮整个网关。网关可以配置连接池。
(3) Load Shedder，设定接口在单位时间里只能接受有限个并发调用，否则 Reject && Fallback
(4) Fallback，能对接口进行设定回退的策略：Fail Fast / Default / Backup
(5) 以上容错逻辑封装于 make_request 逻辑中
j) 环境发现模块，可以发现特定环境的服务
k) 动态配置模块，可以在运行时动态调整一些参数与配置
l) 追踪模块，可以收集 Backend API 的调用
m) 日志模块，收集运行时数据
n) 依赖描述，在接口声明时需要描述依赖的 Backend Service / API
o) 序列化与反序列化模块
p) 压力测试模块，可以测量系统负载进而调优用于熔断/限流/容错的配置

a) NodeJS
(1) 好处
i) 上手快
ii) 启动快
iii) 大量成熟的API Management企业使用NodeJS
(2) 坏处
i) 前方有坑
ii) Dependency Hell
(a) 依赖多
(b) 依赖树不可靠
iii) 社区代码质量偏低
b) Python
(1) 好处
i) 语言成熟
ii) 团队大都比较熟悉
(2) 坏处
i) 前方有坑
c) Java
(1) 好处
i) 语言成熟，稳定
ii) 社区有大量企业级的基础设施与库可以使用
iii) 有现成的一揽子方案可以使用(Spring Cloud)
(2) 坏处
i) 学习曲线
d) 结论
(1) 综合开发速度/社区/熟悉程度
i) 参考 NodeJS 提供的语法糖，改进开发效率
